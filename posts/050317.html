What is Git and Github?

Git is an open source version control system, or VCS. It is used to keep track of and store changes, or commits, made to a working directory or repository. Different from other VCS’s, Git thinks of these changes as snapshots of your whole directory over time as opposed to individual changes over time. It along with GitHub, essentially an online version of Git, provide a very powerful tool to developers for organizing a project in a clear and concise manner.

Terms

status - checks to if there are any differences between your last saved state and the current state of both your local copy and the remote copy

add - used to specify which files, one to many, to track for any changes in the active directory

commit - saves any tracked files to a new snapshot. A message, usually a short explanation of the changes made, can be added to the commit by adding the “-m” flag.

push - takes any commits made locally and saves them to the remote copy

clone - used to copy a remote repository to your computer while syncing the directories, local and remote, so that a new copy does not have to be made every time a change is made

fetch - obtains any changes made to the remote copy without changing your local files. these differences can then be compared before making any changes locally

branch - used to make an exact copy of the current branch, usually done from the master branch, so that changes can be made without disrupting a working program. Also used to refer to the different copies themselves not just the process of making the copy.

merge - combines two copies together. this can be done to branches or the changes obtained through a “fetch” pf the remote copy.

pull - the same as fetch, but also immediately does a merge.

fork - a personal copy of another users repository on GitHub. This allows you to work or play with their code without affecting the original copy.

pull request - a request by you to incorporate the changes made to your fork of a repository back into the original copy. The owner can then accept or reject those changes made.

Creating A Repository

You can start your own repository either locally or remotely, as in on GitHub, but for the purposes of this guide we are going to focus on starting it on GitHub. This is because the process is very similar on either, but starting on GitHub is a bit more straightforward. So to start you are going to click the plus button in the top right of any GitHub page, yes you need to be logged in. On the page that is presented to you, you have a field to name your repository, a description field, and if you want you can create a readme. You need to pay for private repositories so you are probably going to want to leave that option as public. Then from the next page you will want to follow the directions for "...create a new repository on the command line." And that's it. You have made your very first git repository and linked it with a remote copy.

Working With A Repository That Is Not Your Own

So you found some code that you would like to play around with, now what? First off start by clicking that "Fork" button that should be in the top right of your screen. This will make a copy of it to your GitHub account that you can mess around with as you see fit. Once the fork is done you should see a "clone or download" button in green. Click that and then copy over the link. In your terminal first make sure you are in a directory that you want to create this repository, then type "git clone " and paste the link you copied, git return and the repository should be copied to a new directory inside the one you are in. Go ahead and open it up and have fun.

Now that you have made some changes and you like what you have done you are going to want to save or commit those changes. Start by adding these files to be tracking by typing "git add" and then any file names or "git add -A" to track all files that have been changed. Then you want to make your actual commit by "git commit -m 'Some message here'". Now you can push these changes up to your copy of the repository by doing a "git push" or "git push origin 'branch name'". And now you have modified someone else's code and saved it remotely to your copy.

If this is code that you are really proud of AND you think it is something that the owner could use you can preform a pull request on GitHub. This should be located on the left side of your screen about half down. Give them some awesome information about what you did and submit your request. Congratulations you are now a collaborator on GitHub!
